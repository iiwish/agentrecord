# AgentRecord 产品计划

## 一句话定位

AgentRecord 是 AI 时代的本地工作记录层：把用户和各类 AI Agent 协作产生的真实工作痕迹，转化为可审计、可撤销、可分享、可被未来 Agent 使用的个人 AI 工作画像。

## 为什么不是先做招聘决策画像

招聘评分需要跨用户、跨岗位、跨公司场景的校准数据。当前阶段如果直接输出“候选人分数”，很容易变成自嗨高分，长期会损害可信度。

更稳的起点是用户自己的 AI 工作画像：

- 数据来自用户本地真实 Agent 使用记录。
- 先描述“怎么工作”，不是判断“值不值得录用”。
- 所有结论都绑定证据等级、置信度和缺失项。
- 分享页可以服务招聘、合作、开源声誉、个人复盘，但不把招聘决策作为默认产品。

招聘可以作为后续可选视图：招聘者看到的是证据摘要、风险点、面试问题，而不是一个绝对分数。

## 核心用户

第一批用户应该是高频 AI Agent 使用者：

- 使用 Codex、Claude Code、opencode、OpenClaw、Hermes 的开发者和产品型创作者。
- 想让未来 Agent 更懂自己的用户。
- 想展示 AI-native 工作方式但不愿公开原始对话和代码的人。
- 想复盘自己如何使用 Agent、如何验证、如何交付的人。

## 核心价值

### 给用户

- 看清自己在 AI 协作中的真实工作模式。
- 把零散 Agent 使用记录变成稳定资产。
- 生成可分享但不泄露隐私的个人 AI 工作页。
- 生成可给未来 Agent 使用的上下文包。

### 给后来接手的 Agent

- 了解用户偏好的沟通方式、交付标准、验证方式和风险偏好。
- 少问重复问题，少犯用户已经纠正过的错误。
- 更快进入用户的真实工作风格。

### 给外部读者

- 看到证据支持的工作身份，而不是主观简历描述。
- 看到用户是否会拆任务、用工具、验证结果、审查风险、管理范围。
- 可以质疑每个结论，因为结论下面有证据和置信度。

## 产品边界

默认不做：

- 不默认输出招聘分数。
- 不做公开排行榜。
- 不认证“谁更强”。
- 不上传原始对话。
- 不出售原始 trace。

默认要做：

- 本地生成。
- 可重复运行。
- 可审计证据。
- 可分享静态页面。
- 可给 Agent 使用的上下文。
- 多 Agent 适配。

## 顶级项目应具备的产品感

AgentRecord 不能只是一个报表工具。它要让用户第一次打开页面时就产生“这是我的 AI 工作护照”的感觉。

第一屏应该回答：

- 这个人和 AI Agent 协作时最像什么角色？
- 这些结论来自多少时间跨度、多少 Agent 活动、多少证据？
- 哪些能力是强证据，哪些只是弱信号？
- 这个页面为什么可信，隐私边界在哪里？

文案要有传播性，但不能牺牲可信度。可以使用类似：

- Proof of work, not vibes.
- Your AI work record, locally owned.
- Not a prompt history. A work identity.
- 不是提示词历史，是工作方式的证据。
- AI 协作留下的，不只是 token，还有工作画像。

## 商业化路径

先开源 CLI，建立信任和默认入口。

后续商业化可以来自：

- 托管个人页、自定义域名、访问控制。
- 签名快照和可验证 provenance。
- 团队内部 AI 工作画像和能力发展。
- 企业级隐私策略、脱敏规则、审计。
- 高级 adapter 和私有工具集成。

招聘相关服务必须等到证据模型和校准样本足够后再做，否则会破坏项目的长期可信度。
